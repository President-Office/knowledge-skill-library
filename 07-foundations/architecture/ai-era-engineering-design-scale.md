# AI 时代的工程设计尺度

## 核心判断

AI 让实现变便宜了，但没有让错误抽象变便宜。

过去很多团队会用“时间太赶”作为理由，跳过 [[solid]]、[[object-oriented-abstraction]]、[[boundaries]]、[[abstractions]]、测试和重构。AI 可以承担更多样板代码、重复实现和局部改造工作，这意味着我们更少有理由忽略程序本身的设计质量。

但这不等于应该提前设计一套复杂架构。

一句话理解：

**AI 时代更应该设计，但设计的重点是把确定复杂性处理好，而不是为想象中的变化提前支付复杂度。**

## AI 改变了什么

AI 最直接降低的是实现成本：

- 生成 CRUD、DTO、接口、测试样板更快。
- 根据已有模式补齐相似代码更快。
- 查找调用链、解释代码、迁移结构更快。
- 小范围重构和文档化更容易。

这会改变工程节奏：过去被“赶时间”挤掉的设计、测试、重构、复盘，现在应该重新进入工作流。

但 AI 没有自动解决这些问题：

- 模块边界是否合理。
- 数据模型是否稳定。
- 元数据、权限和关系是否能被机器读取。
- 业务不变量是否被保护。
- 抽象是否来自真实变化点。
- 事务和权限是否正确。
- 人是否还能读懂和维护。

所以 AI 不是替代设计，而是释放出更多时间和能力，让设计重新变得可执行。

进一步看，AI 时代的软件系统不只要让人能读懂，也要让机器能读懂。详见：[[machine-readable-software-systems]]。

## 怎么把握度

可以用四条原则判断：

| 原则 | 含义 |
| --- | --- |
| 先设计边界，不急着设计层级 | 模块、数据、权限、事务边界要认真想；继承树、模式堆叠可以晚一点 |
| 先抽象真实变化，不抽象想象变化 | 已经确定会多实现、多策略、多渠道时才建立抽象 |
| 先保证可理解，再追求可扩展 | 简单清楚的普通类，常常胜过没人敢碰的复杂模式 |
| 先让重构变容易，不追求一步到位 | 好设计让后续拆分、替换、迁移成本低 |

更短的判断是：

**不要为没有证据的变化提前支付复杂度；但必须为已经确定的复杂性建立边界。**

## 什么值得提前设计

越难回滚、越影响数据、越影响业务一致性的地方，越值得提前设计。

| 领域                     | 为什么要提前想清楚                 |                          |
| ---------------------- | ------------------------- | ------------------------ |
| [[database]]                    | 表、主键、唯一约束、状态字段进入生产后迁移成本高 |
| 业务边界                   | 边界错了，代码、表、接口、权限都会被牵连      |                          |
| 权限模型                   | 权限散落后很难审计，也容易产生安全问题       |                          |
| [[database-transaction]]                      | 支付、库存、账户、订单等场景不能靠事后补救    |
| 状态机                    | 状态流转一旦散落在多个地方，规则会快速失控     |                          |
| 外部集成边界                 | 支付、短信、对象存储、第三方 API 需要隔离变化 |                          |
| 测试边界                   | 没有测试，AI 生成和重构会放大不确定性      |                          |

这些地方不是“过度设计”，而是基本工程卫生。

## 什么可以先简单

可逆、低风险、变化还不明确的地方，可以先保持简单。

| 领域 | 为什么可以先简单 |
| --- | --- |
| 只有一个实现的接口 | 如果没有替换价值，接口只会增加阅读成本 |
| 复杂继承层级 | 业务模型不稳定时，继承树容易固定错误关系 |
| 过早插件化 | 没有真实扩展方时，插件系统往往只是复杂度 |
| 过早微服务化 | 服务边界和运维成本比代码拆分更重 |
| 临时管理工具 | 低风险内部工具可以先满足真实流程 |
| 页面样式和交互细节 | 不影响核心业务模型时可以迭代调整 |

这里的“简单”不是随便写，而是保持清晰、可读、可测试，给后续演进留空间。

## 设计 20%，实现 60%，整理 20%

一个实用比例是：

```text
设计 20% -> 实现 60% -> 整理 20%
```

过去常见的问题是：为了赶进度，直接把前后的 20% 砍掉，只剩实现。AI 让实现阶段更快，节省出来的时间应该回到：

- 需求拆边界。
- 数据建模。
- 数学建模。
- 写测试。
- 整理命名和目录。
- 补充知识库。
- 复盘为什么这样设计。

但也不能把节省出来的时间全部拿去“造架构”。否则只是从赶工式混乱，变成架构式混乱。

## 判断是否过度设计

可以问这些问题：

| 问题 | 如果答案是“否”，可能过度设计 |
| --- | --- |
| 这个抽象对应真实变化点吗？ | 可能只是为了看起来高级 |
| 现在至少有两个实现、策略或调用方吗？ | 可能抽早了 |
| 新同事能在较短时间内理解吗？ | 可能认知成本过高 |
| 不引入这个设计会造成明确风险吗？ | 可能没有必要 |
| 未来删除这个设计容易吗？ | 如果不容易，要更谨慎 |
| 它是否保护了数据、事务、权限或核心业务规则？ | 如果没有，价值要重新评估 |

过度设计的本质不是“代码多”，而是为不确定收益引入了确定成本。

## 判断是否设计不足

也可以反过来问：

| 问题 | 如果答案是“是”，可能设计不足 |
| --- | --- |
| 一个需求变化要改很多模块吗？ | 边界可能不清 |
| 业务规则散落在 Controller、SQL、前端和脚本里吗？ | 职责可能混乱 |
| 数据库没有约束，只靠代码约定吗？ | 数据模型可能脆弱 |
| 支付、权限、事务等逻辑没有测试吗？ | 风险不可控 |
| 新增一种实现要改大量旧代码吗？ | 抽象或依赖方向可能有问题 |
| AI 生成的代码没人审边界和不变量吗？ | 实现速度正在放大设计缺陷 |

设计不足的本质不是“代码简单”，而是核心复杂性没有被表达和保护。

## 和敏捷的关系

敏捷不是不设计。敏捷真正反对的是把大量未经验证的假设一次性固化。

在 AI 时代，更合理的敏捷是：

- 用较短周期验证真实需求。
- 对已验证的稳定部分建立清晰边界。
- 对未验证的部分保持可修改。
- 让测试和重构成为迭代的一部分。
- 让 AI 处理实现重复，让人判断边界和取舍。

所以敏捷和设计并不矛盾。好的敏捷需要好设计，否则每次迭代都会变成堆补丁。

## 和 OOP / SOLID 的关系

[[object-oriented-abstraction]]、[[solid]]、[[interface-class-abstract-class]] 这些思想在 AI 时代更重要，但使用方式要更克制。

- [[single-responsibility-principle]] 帮助识别变化原因，不是把每个函数都拆成类。
- [[open-closed-principle]] 适合真实扩展点，不是所有地方都插件化。
- [[liskov-substitution-principle]] 约束继承和实现关系，防止抽象不可靠。
- [[interface-segregation-principle]] 防止接口变成大而全的负担。
- [[dependency-inversion-principle]] 让核心业务依赖稳定抽象，而不是依赖基础设施细节。

这些原则的价值不是让代码“显得专业”，而是降低变化成本。

## 在后台项目中怎么用

| 场景 | 建议 |
| --- | --- |
| 高考志愿模块 | 先划清专业查询、院校查询、分数查询、职业查询、志愿生成的业务边界 |
| 支付模块 | 抽出支付接口是合理的，因为支付渠道和测试替身是真实变化点 |
| 数据导入 | 如果多个导入任务共享稳定流程，可以用抽象类或模板方法 |
| 普通 CRUD | 不要为了一个实现强行接口化，但要保证数据约束和权限清楚 |
| 报表分析 | 区分 [[oltp]] 交易数据和 [[olap]] 分析链路，不要把所有查询压在主库 |
| 多数据库支持 | 不只是换驱动，要审 SQL 方言、分页、事务和索引差异 |
| 志愿推荐、退款、积分、排班 | 先做 [[mathematical-modeling]]，明确变量、约束、目标、单位和测试 |
