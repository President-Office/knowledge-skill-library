# LSP 里氏替换原则

## 核心定位

LSP 是 Liskov Substitution Principle，通常译为“里氏替换原则”。

一句话理解：

**子类型应该能替换父类型，并且不破坏调用方依赖的行为。**

它关心的不是继承语法是否正确，而是替换之后系统行为是否仍然符合契约。

## 它解决什么问题

如果子类不能真正替代父类，调用方就必须知道具体子类的特殊规则，抽象会失效。

典型坏信号：

- 子类重写方法后抛出“不支持该操作”。
- 父类承诺成功返回，子类却悄悄改变异常语义。
- 父类要求幂等，子类实现不幂等。
- 父类表示“所有存储都支持删除”，某个实现却不能删除。

## 后台项目里的判断

| 场景 | LSP 追问 |
| --- | --- |
| 多数据库适配 | [[mysql]]、[[postgresql]]、[[oracle-database]] 实现是否遵守同一查询和事务语义 |
| 多存储适配 | 本地、OSS、S3 是否都满足上传、下载、删除契约 |
| 支付渠道 | 不同支付实现是否对成功、失败、回调、退款有一致语义 |
| 权限策略 | 替换策略后，调用方是否仍能得到稳定的权限判断结果 |
| 测试替身 | mock / fake 是否足够接近真实实现的关键行为 |

## 和继承、多态的关系

[[inheritance]] 给了类型替换的形式，[[polymorphism]] 让替换发生在运行时，LSP 判断这种替换是否可靠。

如果违反 LSP，多态表面上存在，调用方实际上还是被迫依赖具体实现。

## 常见误区

- LSP 不是只适用于继承树，接口实现也同样适用。
- 通过 if 判断具体类型来绕过问题，通常说明抽象契约有缺陷。
- 子类功能更多没问题，但不能破坏父类已经承诺的行为。
