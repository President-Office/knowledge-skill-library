# ISP 接口隔离原则

## 核心定位

ISP 是 Interface Segregation Principle，通常译为“接口隔离原则”。

一句话理解：

**接口不应该强迫调用方依赖自己不用的方法。**

它关注接口的能力面是否过大。接口太大，调用方会被迫理解和依赖不需要的能力，实现方也会被迫实现不适合自己的方法。

## 它解决什么问题

大接口会造成这些问题：

- 实现类被迫实现空方法或抛异常。
- 调用方看到一堆自己不用的方法，理解成本上升。
- 一个能力变化，影响所有依赖这个大接口的调用方。
- 权限、SDK、Service API 越长越难维护。

ISP 追问的是：调用方真正需要的是哪一小组能力？

## 后台项目里的判断

| 场景       | 更符合 ISP 的做法                   |
| -------- | ----------------------------- |
| 用户服务     | 查询用户、修改用户、分配角色可以拆成更清晰能力       |
| 文件服务     | 上传、下载、删除、签名 URL 不一定强迫所有实现都支持  |
| 支付服务     | 支付、退款、查询、关闭订单可以按渠道能力拆分        |
| SDK      | 客户端只暴露当前场景需要的 API 面           |
| 前端 hooks | 一个 hook 不应该同时处理列表、详情、编辑、导出、权限 |

## 和 SRP 的关系

[[single-responsibility-principle]] 关注模块职责，ISP 关注接口能力面。两者都在降低变化扩散。

一个接口如果同时服务太多调用方，往往也说明职责不清。

## 常见误区

- 接口隔离不是把每个方法拆成一个接口。
- 小接口也需要有稳定语义，不能只按技术动作机械拆分。
- 如果所有调用方总是一起使用一组方法，把它们放在一个接口里可能更清楚。
