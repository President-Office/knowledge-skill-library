# 继承

## 核心定位

继承用来表达类型之间的“是一种”关系，也可以复用或扩展父类行为。

一句话理解：

**继承可以承载抽象层级，但继承不是抽象本身，也不是复用代码的默认答案。**

## 和抽象的关系

[[abstractions]] 关心稳定契约；继承可以把共同契约放在父类或抽象类里，让子类提供差异实现。

例如：

```text
PaymentClient
  -> WeChatPayClient
  -> AliPayClient
```

但很多好的抽象并不依赖继承，而是依赖接口、组合、函数参数、模块边界或协议。

## 什么时候适合继承

| 判断问题 | 合理信号 |
| --- | --- |
| 是否是稳定的“是一种”关系 | 子类确实可以替代父类 |
| 父类是否有清晰契约 | 调用方依赖父类时不需要知道具体子类 |
| 子类差异是否可控 | 子类主要替换差异点，而不是推翻父类行为 |
| 层级是否浅 | 继承链不会深到难以理解 |

## 继承的风险

- 父类变化可能影响所有子类。
- 子类为了复用代码而继承父类，容易继承不需要的行为。
- 继承层级太深时，读一个方法要在多层父类之间跳转。
- 如果子类不能替代父类，会违反 [[solid]] 里的 [[liskov-substitution-principle]]。

## 常见替代方式

- 接口：只表达契约，不继承实现。
- 组合：把可复用能力作为成员对象注入。
- 策略模式：把变化的算法或规则独立出来。
- Adapter：隔离外部系统差异。
