# 封装

## 核心定位

封装是把对象的内部状态和实现细节藏起来，只暴露稳定、合法的操作入口。

一句话理解：

**封装不是把字段改成 `private`，而是把“外部能做什么、不能做什么”表达清楚。**

## 和抽象的关系

[[abstractions]] 关心调用方应该依赖什么稳定概念；封装负责把这个稳定概念背后的细节藏起来。

例如支付能力可以暴露一个 `pay()` 方法，调用方不需要知道底层是微信支付、支付宝、Stripe，还是银行转账。

## 封装保护什么

| 保护对象 | 含义 | 例子 |
| --- | --- | --- |
| 内部状态 | 不允许外部随意改对象内部数据 | 订单金额、库存数量、账户余额 |
| 业务不变量 | 对象始终保持合法状态 | 已支付订单不能随意改金额 |
| 实现细节 | 底层实现可以替换 | 数据库、缓存、第三方接口 |
| 操作顺序 | 外部只能通过合法流程修改状态 | 订单先创建，再支付，再发货 |

## 常见误区

- 只有 getter/setter 不等于封装；如果所有字段都能读写，业务规则仍然暴露在外部。
- 封装不是隐藏一切；必要的业务能力必须用清晰方法暴露出来。
- 封装不能只按技术字段设计，还要表达业务动作。

## 在项目中怎么观察

| 场景 | 观察点 |
| --- | --- |
| 订单服务 | 是否通过业务方法修改状态，而不是直接改字段 |
| 支付模块 | 是否隐藏签名、回调、渠道参数等细节 |
| 权限模块 | 是否把角色、菜单、租户、数据权限规则封装在统一入口后面 |
| 文件存储 | 是否隐藏本地磁盘、OSS、S3 等实现差异 |
