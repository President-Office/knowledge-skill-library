# 数学建模

## 核心定位

数学建模是把现实业务问题转换成变量、约束、目标、公式、算法和验证规则的过程。

一句话理解：

**数学建模不是先找算法，而是先把业务问题表达成可计算、可验证、可解释的模型。**

在业务系统里，数学建模经常出现在：

- 价格、退款、佣金、积分、补贴。
- 库存、容量、名额、配额。
- 排课、排班、路径、调度。
- 高考志愿推荐、风险评分、分数位次匹配。
- 报表指标、转化率、留存率、ROI。

## 数学建模解决什么

| 问题 | 建模要回答什么 |
| --- | --- |
| 业务要做什么决策 | 推荐、排序、分配、计算、预测、约束还是优化 |
| 输入是什么 | 用户数据、业务记录、配置参数、时间、状态 |
| 输出是什么 | 金额、分数、方案、结果集、解释、告警 |
| 哪些规则必须满足 | 不变量、边界、权限、容量、合法状态 |
| 优化目标是什么 | 成本最低、收益最高、风险最低、覆盖最大、体验最好 |
| 精度如何处理 | 单位、舍入、误差、金额精度、日期边界 |

没有建模时，业务规则通常会散落在 Controller、SQL、前端和脚本里。短期能跑，长期很难解释和验证。

## 基本构件

| 构件 | 含义 | 例子 |
| --- | --- | --- |
| 变量 | 会变化、需要计算或选择的量 | 分数、排名、退款金额、选择的院校 |
| 参数 | 外部配置或固定输入 | 费率、阈值、批次、省份、年份 |
| 单位 | 数值代表什么量纲 | 元、分、天、公里、人次、百分比 |
| 约束 | 必须满足的条件 | 退款不超过实付金额，名额不能为负 |
| 目标函数 | 要优化的方向 | 风险最小、匹配度最高、成本最低 |
| 不变量 | 任意时候都必须成立的规则 | 余额不能为负，库存不能超卖 |
| 精度 | 数值计算和舍入规则 | 金额用两位小数，除法指定舍入 |
| 算法 | 求解模型的方法 | 公式、排序、图算法、线性规划、约束求解 |

建模的关键是先把这些说清楚，再决定用普通公式、查询、规则引擎、图算法还是优化求解器。

## 一个建模卡片模板

复杂业务规则进入代码前，可以先写一张模型卡：

| 项目 | 内容 |
| --- | --- |
| 名称 | 模型叫什么 |
| 业务目的 | 它帮助谁做什么决策 |
| 公式/模型版本 | 以后规则变化时如何追踪 |
| 输入 | 需要哪些数据 |
| 输出 | 产生什么结果 |
| 单位 | 每个数值的单位是什么 |
| 精度和舍入 | 金额、比例、分数如何处理 |
| 约束 | 什么情况必须禁止 |
| 不变量 | 任何时候都必须成立的规则 |
| 目标函数 | 如果是优化问题，优化什么 |
| 异常情况 | 数据缺失、无解、越界、重复如何处理 |
| 测试用例 | 正常、边界、异常、历史样例 |
| 审计字段 | 输入快照、输出快照、模型版本、计算时间 |

这张卡不是文档负担，而是防止业务公式变成“谁也说不清的代码”。

## 和数据建模的区别

[[database]] 关注业务事实如何长期保存。  
数学建模关注业务事实如何被计算、约束、优化和解释。

| 主题 | 关注点 |
| --- | --- |
| 数据建模 | 表、字段、关系、约束、索引、事务 |
| 数学建模 | 变量、参数、单位、公式、约束、目标、算法 |

两者必须配合。比如高考志愿推荐中：

- 数据建模负责保存院校、专业、分数、位次、招生计划。
- 数学建模负责计算匹配度、风险等级、排序规则和志愿方案。

## 和 OOP / 领域模型的关系

数学模型需要在代码里有明确落点。

好的做法是：

- 用 [[class]] 或值对象表达输入和输出。
- 用领域服务表达纯计算和规则。
- 用 [[database-transaction]] 包住会改变业务状态的流程。
- 用 [[orm]] / Mapper 读取事实数据，但不要把核心公式塞进 Mapper。
- 用测试验证不变量和边界条件。

不要把数学模型直接写成散落的 if/else、SQL 片段和前端判断。那样既难复用，也难证明正确。

## 常见模型类型

| 类型 | 适用场景 |
| --- | --- |
| 确定性公式 | 价格、退款、积分、佣金、评分 |
| 规则模型 | 资格判断、状态流转、准入条件 |
| 排序模型 | 推荐列表、志愿排序、搜索排序 |
| 约束模型 | 名额、库存、容量、时间窗口 |
| 图模型 | 路径、路线、依赖、关系网络 |
| 优化模型 | 排班、调度、资源分配、路径规划 |
| 统计模型 | 指标、趋势、风险分层、转化分析 |

不是所有模型都需要复杂算法。很多业务系统最需要的是把普通公式、边界和单位写清楚。

## 高考志愿例子

志愿生成不是简单 CRUD，它天然需要数学建模。

| 建模要素 | 高考志愿中的例子 |
| --- | --- |
| 输入 | 考生分数、位次、省份、科类、选科、偏好、预算 |
| 数据事实 | 历年分数线、专业录取位次、招生计划、院校层次 |
| 变量 | 可选择的院校、专业、志愿顺序 |
| 约束 | 批次、选科要求、省份招生、专业限制 |
| 目标 | 提高录取概率，同时兼顾院校、专业、地域偏好 |
| 输出 | 冲稳保分层、志愿方案、风险解释 |
| 不变量 | 不推荐不满足选科或批次限制的专业 |

如果这些没有明确模型，代码很容易变成大量特殊条件拼接，最后没人能解释为什么推荐这个结果。

## 验证方式

数学建模必须可验证。

| 验证方式 | 适合检查什么 |
| --- | --- |
| 示例测试 | 真实业务样例是否算对 |
| 边界测试 | 0、最大值、缺失值、过期、重复、无解 |
| 不变量测试 | 金额不为负、库存不超卖、推荐结果满足约束 |
| 回归测试 | 规则改动后历史样例是否按预期变化 |
| 审计记录 | 重要决策能否追溯输入、输出和模型版本 |

AI 可以快速生成实现，但模型是否正确，仍然要靠明确的业务定义和测试验证。

## 常见误区

- 把数学建模等同于高级算法：很多建模只是把普通业务规则说清楚。
- 先选算法再理解业务：容易做出精致但无效的系统。
- 不写单位和精度：金额、比例、时间一旦混乱，错误很难排查。
- 把公式写进 Controller 或 SQL：短期方便，长期难测难改。
- 只看平均值，不看边界和异常：真实业务经常死在边界条件。
- 没有模型版本：公式变化后无法解释历史结果。
