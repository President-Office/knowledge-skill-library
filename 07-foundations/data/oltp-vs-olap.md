# OLTP vs OLAP

## 核心区别

一句话理解：

**[[oltp]] 负责记录业务事实，[[olap]] 负责从业务事实中提炼洞察。**

| 类型       | 关注点      | 典型场景   | 常见数据库           | 举例                                                                         |
| -------- | -------- | ------ | --------------- | -------------------------------------------------------------------------- |
| [[oltp]] | [[OLTP]] | 处理日常交易 | 下单、支付、注册、库存扣减   | [[mysql]]、[[postgresql]]、[[oracle-database]]、[[sql-server]]、[[SQL Server]] |
| [[olap]] | [[OLAP]] | 分析大量数据 | 报表、BI、指标看板、趋势分析 | [[clickhouse]]                                                             |

## 访问模式对比

| 维度 | OLTP | OLAP |
| --- | --- | --- |
| 数据操作 | 高频新增、修改、查询单条或少量记录 | 扫描大量历史数据并聚合 |
| 查询方式 | 按主键、订单号、用户 ID 查询 | 按时间、地区、渠道、品类、客户类型聚合 |
| 数据模型 | 规范化表结构，强调一致性 | 维度、指标、宽表、预聚合 |
| 性能目标 | 低延迟交易响应 | 大规模统计查询吞吐 |
| 数据来源 | 直接承载业务写入 | 通常从业务库同步、清洗、汇总 |

## 常见链路

```text
业务操作 -> OLTP 数据库 -> 同步/ETL -> OLAP 数据库 -> 报表/BI/看板
```

例如：

- MySQL 订单表记录每一笔订单，这是 OLTP。
- ClickHouse 按月份、地区、品类统计销售额，这是 OLAP。

## 学习重点

- 先判断系统是在记录事实，还是在分析事实。
- 不要把报表慢简单归因于数据库选型，数据模型、同步链路、指标口径和预聚合同样重要。
- 不要把 OLAP 数据库当作普通业务主库，交易一致性和分析吞吐是不同目标。
