# 事务

## 核心定位

事务是一组必须作为整体成功或失败的数据库操作。

一句话理解：

**事务回答“这些操作要不要一起成功、一起失败”。**

例如支付场景：

```text
创建支付单
扣减账户余额
记录资金流水
更新订单状态
```

这些操作如果只成功一部分，业务数据就会矛盾。

## ACID

事务通常用 ACID 描述：

| 特性 | 含义 |
| --- | --- |
| Atomicity 原子性 | 要么全部成功，要么全部失败 |
| Consistency 一致性 | 事务前后数据满足规则 |
| Isolation 隔离性 | 并发事务之间不能随意互相污染 |
| Durability 持久性 | 提交后的结果应该可靠保存 |

学习事务时，不要只背 ACID，要看业务流程哪里需要一致性边界。

## 后台项目里的事务边界

事务通常出现在 Service 层，而不是 Controller 层。

应该放进一个事务的操作：

- 创建订单和订单明细。
- 支付成功后更新订单、支付单、资金流水。
- 志愿方案确认时写入方案和方案项。
- 审批通过时更新状态并记录审批日志。

不一定适合放进同一个数据库事务的操作：

- 调用第三方接口。
- 发送短信或邮件。
- 大文件上传。
- 长时间算法计算。

这些通常需要幂等、消息、补偿或最终一致性配合。

## 事务和 OLTP

[[oltp]] 系统非常依赖事务，因为它负责记录日常业务交易。

[[olap]] 系统更关注大量历史数据分析，通常不会把复杂交易一致性作为核心目标。

## 常见误区

- 事务不是越大越好，事务越长，锁和并发成本越高。
- 分布式事务不是默认答案，很多场景更适合幂等和最终一致性。
- 事务只能保护数据库内的操作，不能自动保护外部接口调用。
