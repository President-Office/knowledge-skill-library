# OLTP

## 核心定位

OLTP 是 Online Transaction Processing，中文常译为“联机事务处理”。它关注的是业务系统中的日常交易：新增、修改、查询少量业务记录，并保证这些操作正确、及时、可回滚。

一句话理解：

**OLTP 负责记录业务事实，[[olap]] 负责从业务事实中提炼洞察。**

## 典型问题

OLTP 适合处理这类问题：

- 用户注册、登录、修改资料。
- 创建订单、支付订单、取消订单。
- 库存扣减、库存回滚。
- 会员余额充值、扣款、退款。
- 审批流提交、同意、驳回。
- 后台新增用户、分配角色、修改菜单权限。

## 关键要求

| 要求   | 为什么重要                 |
| ---- | --------------------- |
| 事务   | 多个写操作要么一起成功，要么一起失败    |
| 一致性  | 订单、支付、库存、账户不能互相矛盾     |
| 低延迟  | 用户操作和业务接口需要快速响应       |
| 并发控制 | 多人同时操作时不能产生脏数据        |
| 可追溯  | 核心业务变更通常要保留日志、流水或审计记录 |

## 和 OLAP 的关系

OLTP 负责记录业务事实，[[olap]] 负责从业务事实中提炼洞察。两者的完整对比放在独立卡片：[[oltp-vs-olap]]。

## 为什么 OLTP 通常先于 OLAP

分析系统不能凭空产生洞察。通常先有 OLTP 系统记录业务事实，再把数据同步到 OLAP 系统做统计。

```text
业务操作 -> OLTP 数据库 -> 同步/ETL -> OLAP 数据库 -> 报表/BI/看板
```

这个链路的重点不是“先买一套分析数据库”，而是先让业务事实被正确记录，再考虑如何同步、汇总和分析。

## 不要误解

- OLTP 不等于“用了关系型数据库”，它强调的是交易处理目标。
- OLTP 数据库可以被查询做简单报表，但大规模聚合分析通常应该进入 OLAP 链路。
- 分布式事务不是 OLTP 的默认答案，很多问题更应该通过清晰边界、幂等和最终一致性解决。
