# 索引

## 核心定位

Index，中文通常叫索引，是数据库为了更快查找数据而维护的辅助结构。

一句话理解：

**索引回答“怎样更快找到需要的行”。**

没有索引时，数据库可能需要从头到尾扫描大量行。  
有合适索引时，数据库可以先在索引结构里定位候选记录，再回到表中读取完整数据。

可以把它粗略理解成书的目录或术语索引：

```text
没有索引：从第一页翻到最后一页
有索引：先查目录/索引页，再跳到目标页
```

这个比喻不完全等同于数据库内部实现，但能帮助建立第一直觉：索引用额外结构换取查询速度。

## 索引为什么重要

后台系统里的很多性能问题，本质不是代码循环慢，而是数据库在扫描太多数据。

例如：

```sql
select *
from admission_score
where province = '陕西'
  and year = 2025
  and batch = '本科一批'
  and school_id = 123;
```

如果 `admission_score` 有几百万行，而这些查询条件没有合适索引，数据库可能要扫描大量记录。  
如果建立了合适的联合索引，数据库就能更快缩小范围。

所以索引连接了三件事：

- [[sql]] 的 `where`、`join`、`order by`、`group by`。
- [[database-index]] 本身的结构和字段顺序。
- 真实业务查询的频率、数据量和选择性。

## 常见索引场景

| 场景 | 例子 |
| --- | --- |
| 按主键查询 | `where id = ?` |
| 按唯一字段查询 | `where mobile = ?` |
| 按外键查询明细 | `where school_id = ?` |
| 按状态筛选 | `where status = ?` |
| 按时间排序 | `order by create_time desc` |
| 联合查询 | `where province = ? and year = ? and batch = ?` |

高考录取数据中，常见索引可能围绕：

```text
province + year + batch + school_id + major_id
```

这类索引不是凭空设计出来的，而是从真实查询路径反推出来的：

- 用户经常按省份和年份查分数。
- 系统经常按院校和专业定位历史录取数据。
- 列表页经常按批次、类别、分数段过滤。

## 常见索引类型

| 类型 | 解决的问题 |
| --- | --- |
| 普通索引 | 加速普通条件查询 |
| 唯一索引 | 加速查询，同时保证字段组合唯一 |
| 主键索引 | 通过主键唯一定位一行 |
| 联合索引 | 用多个字段共同支持查询条件 |
| 覆盖索引 | 查询所需字段都在索引中，减少回表 |
| 全文索引 | 支持文本检索，能力依赖具体数据库 |
| 倒排索引 | 搜索引擎常用结构，典型项目是 [[elasticsearch]] |

关系型数据库里最常先理解普通索引、唯一索引、主键索引和联合索引。

## 联合索引

联合索引是后台系统中非常常见、也最容易误用的索引。

例如：

```sql
create index idx_score_query
on admission_score (province, year, batch, school_id);
```

它适合支持类似查询：

```sql
where province = ?
  and year = ?
  and batch = ?
  and school_id = ?
```

联合索引的字段顺序很重要。通常应该把查询中稳定出现、过滤效果明显、能缩小范围的字段放在前面。不同数据库优化器能力不同，但“索引字段顺序要服务真实查询”这个原则长期有效。

## 选择性

选择性可以粗略理解为：一个字段能把数据区分得多细。

| 字段 | 选择性直觉 |
| --- | --- |
| `id` | 很高，每行不同 |
| `mobile` | 很高，通常唯一 |
| `province` | 中等，全国省份有限 |
| `status` | 可能较低，状态值很少 |
| `deleted` | 很低，通常只有 0/1 |

低选择性字段单独建索引，收益可能有限。  
但低选择性字段如果经常和其他字段一起过滤，放进联合索引仍然可能有价值。

## 索引不是越多越好

索引会提升查询，但也有代价：

- 写入、更新、删除时需要维护索引。
- 索引占用存储空间。
- 错误索引可能无法被查询使用。
- 太多索引会增加优化器选择成本和维护成本。

所以索引设计要围绕真实查询。

## 索引和唯一约束

唯一索引既能加速查询，也能保证唯一性。

例如：

```text
tenant_id + mobile
```

可以表达“同一租户下手机号唯一”。

这既是查询需求，也是业务约束。

## 索引和执行计划

判断索引有没有用，不能只看“建了索引没有”，还要看数据库执行计划。

执行计划会告诉你：

- 数据库是否使用了索引。
- 使用了哪个索引。
- 预计扫描多少行。
- 是否发生排序、临时表、回表等额外操作。

在 MySQL 中常见入口是：

```sql
explain
select *
from admission_score
where province = '陕西'
  and year = 2025;
```

执行计划是从“我觉得应该快”走向“数据库实际怎么执行”的关键工具。

## 和搜索索引的区别

关系型数据库索引主要服务结构化查询，例如按 ID、状态、时间、外键过滤。  
搜索引擎索引更擅长全文检索、相关性排序、分词和复杂搜索体验。

例如：

| 需求 | 更常见选择 |
| --- | --- |
| 按订单 ID 查询订单 | 关系型数据库索引 |
| 按手机号查用户 | 唯一索引 |
| 搜索文章标题和正文 | [[elasticsearch]] 倒排索引 |
| 商品关键词搜索和排序 | 搜索索引 + 业务数据库 |

搜索索引通常不应该成为唯一业务事实来源。主数据仍应落在 [[database]] 中。

## 常见误区

- 看到慢查询不要只说“加索引”，还要看 SQL、数据量、选择性、排序、分页方式。
- 联合索引的字段顺序很重要。
- 低区分度字段单独建索引可能收益有限。
- 索引不能替代清晰的数据模型。
- 建了索引不代表一定会被使用，要看执行计划。
- 只为单条慢 SQL 建索引前，要评估写入成本和整体查询模式。
