# 约束

## 核心定位

约束是数据库用来保证数据合法性的规则。

一句话理解：

**约束回答“哪些数据不允许进入数据库”。**

## 常见约束

| 约束 | 作用 | 例子 |
| --- | --- | --- |
| NOT NULL | 字段不能为空 | 学校名称不能为空 |
| UNIQUE | 字段或字段组合必须唯一 | 同一租户下手机号唯一 |
| PRIMARY KEY | 唯一标识一行 | `id` |
| FOREIGN KEY | 保证引用关系存在 | `major.school_id` 必须存在于 `school.id` |
| CHECK | 限制字段取值范围 | 分数不能小于 0 |
| DEFAULT | 设置默认值 | `deleted = false` |

## 为什么不能只靠代码校验

代码校验很重要，但数据库约束仍然有价值：

- 多个服务可能写同一张表。
- 脚本、导入工具、后台管理都可能绕过业务代码。
- 并发情况下，唯一性只靠先查再写容易出错。
- 数据库是长期事实载体，应该保护底线规则。

## 约束和业务规则

不是所有业务规则都适合放进数据库约束。

| 适合数据库约束 | 更适合业务代码 |
| --- | --- |
| 非空、唯一、基础范围 | 复杂审批规则 |
| 主外键关系 | 跨系统状态机 |
| 字段默认值 | 依赖第三方接口的判断 |
| 简单 check | 需要上下文的策略 |

数据库约束保护底线，业务代码表达复杂流程。

## 常见误区

- 没有约束的数据表容易变成垃圾入口。
- 约束不是越多越好，过度约束会增加迁移和导入成本。
- 软删除和唯一约束要特别小心，例如同一手机号删除后是否允许重新注册。
