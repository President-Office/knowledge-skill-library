# 主键

## 核心定位

主键是表中用来唯一识别一行数据的字段或字段组合。

一句话理解：

**主键回答“我如何准确找到这一行”。**

常见主键：

```text
id
school_id
student_id
order_id
```

## 为什么重要

主键影响：

- 数据是否能被稳定引用。
- 外键如何关联。
- 更新和删除是否能定位到正确记录。
- ORM 如何识别对象身份。
- 分页、同步、去重和审计如何实现。

## 自增 ID 和业务 ID

| 类型 | 优点 | 风险 |
| --- | --- | --- |
| 自增 ID | 简单、数据库友好 | 分布式场景和迁移时要小心 |
| UUID / 雪花 ID | 全局唯一，适合分布式 | 可读性和索引局部性可能较差 |
| 业务 ID | 业务可读，例如订单号 | 业务规则变化会影响身份 |

实践中常见做法是：

```text
内部主键：id
业务编号：code / no / sn
```

二者不要混淆。

## 主键和唯一约束

主键一定唯一且通常不能为空，但不是所有唯一字段都应该做主键。

例如：

- 用户手机号可以唯一，但可能会换绑。
- 学校代码可能唯一，但不同数据源可能存在变更或映射问题。
- 订单号通常是业务唯一编号，但内部仍可能有自增或雪花主键。

## 常见误区

- 不要把会变化的业务字段轻易做主键。
- 不要没有主键，后续更新、同步、审计都会困难。
- 联合主键可以表达关系，但在后台开发和 ORM 映射中要考虑复杂度。
